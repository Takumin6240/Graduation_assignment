## 2. 第1章「はじめに」の構成

### 2.1 背景

#### 2.1.1 プログラミング教育の必修化
**文部科学省の方針**[1]:
- 2020年度から小学校でプログラミング教育が必修化
- 学習指導要領において「プログラミング的思考」の育成が明記
- 各教科において年間を通じて計画的に実施

**プログラミング的思考とは**[2]:
- 自分が意図する一連の活動を実現するために
- どのような動きの組合せが必要か
- どのように改善していけばより意図した活動に近づくか
- 論理的に考えていく力

**Scratchの普及**:
- ビジュアルプログラミング言語として小学校で広く採用
- ブロックを組み合わせてプログラムを作成
- 直感的な操作で小学生でも理解しやすい

#### 2.1.2 教育現場の現状と課題

**現状：模倣（丸写し）型の教材が中心**[3]:
- NPO情報セキュリティフォーラムの調査(2021年)によると、多くの小学校で採用されている教材は「完成見本を真似て作る」形式が中心
- 教科書や指導書に示された手順通りに進めることで成果物を完成させる
- 「作品を完成させること」が目的化している傾向

**この方式の問題点**:
1. **試行錯誤のプロセスが欠如**
   - 手順通りに進めれば完成するため、自分で考える機会が少ない
   - なぜそのブロックが必要なのか理解しないまま進む

2. **プログラミング的思考が育ちにくい**[4]:
   - 静岡大学の研究によると、模倣型学習では論理的思考力の向上が見られにくい
   - 「できる」と「わかる」の乖離が発生
   - 理解しなくても完成させることができてしまう

3. **知識の応用力が身につかない**:
   - 見本と異なる問題に遭遇すると対応できない
   - 学習内容が定着せず、応用問題が解けない

### 2.2 問題点の詳細

#### 2.2.1 学習者側の問題
1. **表面的な理解に留まる**
   - プログラムの実行結果を予測する力が養われない
   - ブロックの意味を理解せずに配置している
   - 「できたつもり」になってしまう問題

2. **試行錯誤の経験不足**
   - エラーが出た時の対処方法がわからない
   - 自分で考えて修正する機会が少ない
   - デバッグ能力が育たない

3. **段階的な理解の不足**
   - いきなり複雑なプログラムを作ろうとする
   - 基礎から応用への橋渡しが不十分
   - 発達段階に応じた学習設計が必要[5]

#### 2.2.2 指導者側の問題
1. **個別対応の困難さ**
   - 1クラス30〜40人の児童全員の理解度を把握しきれない
   - 理解度に応じた個別指導が難しい

2. **採点・評価の負担**
   - プログラムの正誤を手動で確認するには時間がかかる
   - どこまで理解できているか客観的に評価しづらい
   - 評価基準があいまいになりがち

3. **効果測定の難しさ**
   - プログラミング的思考が育っているか測定しづらい
   - 学習進捗の可視化が不十分

### 2.3 研究目的

本研究の目的は、**模倣学習の構造から脱却し、試行錯誤のプロセスそのものを学習の中心に据えることで、効果的にプログラミング的思考を育成すること**である。

具体的には以下を達成する:

1. **「考えさせる」学習への転換**
   - 完成見本を模倣するのではなく、自分で考えてプログラムを構築・修正する
   - 試行錯誤を通じて理解を深める学習プロセスの実現

2. **段階的な思考力の育成**
   - 基礎的な理解から応用的な創造へと段階的にステップアップ
   - 発達段階(小学3〜6年生)に応じた適応的な学習支援

3. **教員負担の軽減**
   - 自動採点機能により評価業務を効率化
   - 学習進捗の可視化により効果的な指導を支援

### 2.4 提案概要

上記の目的を達成するため、本研究では**4種類の問題形式を用いた段階的学習システム**を提案する。

#### 2.4.1 4種類の問題タイプ

**1. 穴埋め式問題(fill_blank)**
- **目的**: プログラムの一部を推察させる
- **育成する力**: 基本的なブロックの理解、文脈からの推論
- **例**: 「10歩動かす」プログラムで「○○歩動かす」の○○を埋める

**2. 予測問題(predict)**
- **目的**: コードの実行結果を予測させる
- **育成する力**: プログラムの動作理解、実行フローの把握
- **例**: 提示されたプログラムを見て「スプライトはどこに移動するか」を答える

**3. 間違い探し問題(find_error)**
- **目的**: プログラムの誤りを発見・修正させる
- **育成する力**: デバッグ能力、論理的思考、エラー発見力
- **例**: 意図した動作をしないプログラムのバグを修正する

**4. ミッション型問題(mission)**
- **目的**: 知識を応用して創造的に問題を解決させる
- **育成する力**: 創造力、問題解決能力、知識の統合
- **例**: 「迷路を脱出するプログラムを作りなさい」といった課題

#### 2.4.2 システムの主要機能

1. **.sb3ファイルの自動採点エンジン**
   - Scratchの.sb3ファイルを解析し、自動で採点
   - **変数名の違いを吸収する変数マッピング機能(v2.1)** ← 最大の技術的貢献
   - 柔軟な評価(±10%許容)で小学生の入力誤差を考慮

2. **学年別適応型ヒント機能**
   - 小学3〜6年生の発達段階に応じたヒント提供
   - 学年ごとに表現や難易度を調整

3. **詳細なフィードバック**
   - どこが正解でどこが間違っているか具体的に提示
   - 次のステップへの改善提案

4. **進捗管理とゲーミフィケーション**
   - レベル・経験値システムで学習意欲を維持
   - 学習履歴の記録と可視化

#### 2.4.3 対象学年

**小学3〜6年生**を対象とする。

選定理由[5]:
- ピアジェの発達段階論によると、7〜11歳(具体的操作期)は論理的思考が発達する時期
- 小学3年生以上であれば、プログラミングの基本的な概念を理解できる
- 低学年(1〜2年生)は抽象的な思考が難しいため対象外とした

## 3. 第2章「関連研究」の構成

### 3.1 背景の詳細

#### 3.1.1 プログラミング教育必修化の経緯
**学習指導要領の改訂**[1][2]:
- 2017年に改訂された学習指導要領において、プログラミング教育が初めて位置づけられた
- 「プログラミング的思考」を育成することが明記された
- 各教科等の中で実施するとされ、独立した教科ではない

**実施状況**:
- 2020年度から全面実施
- 算数、理科、総合的な学習の時間などで実施
- Scratchなどのビジュアルプログラミング言語が広く採用

#### 3.1.2 プログラミング的思考(計算論的思考)とは

**文部科学省の定義**[1]:
> 自分が意図する一連の活動を実現するために、どのような動きの組合せが必要であり、一つ一つの動きに対応した記号を、どのように組み合わせたらいいのか、記号の組合せをどのように改善していけば、より意図した活動に近づくのか、といったことを論理的に考えていく力

**4つの要素**(計算論的思考の構成要素):
1. **分解(Decomposition)**: 問題を小さな部分に分ける
2. **パターン認識(Pattern Recognition)**: 共通点や規則性を見つける
3. **抽象化(Abstraction)**: 重要な情報を抽出する
4. **アルゴリズム化(Algorithm Design)**: 手順を論理的に組み立てる

**小学生における重要性**:
- 将来どのような職業に就くとしても、時代を超えて普遍的に求められる力
- 問題解決能力の基礎となる
- 論理的思考力の育成につながる

#### 3.1.3 模倣学習の問題点の詳細

**NPO情報セキュリティフォーラムの調査結果**[3]:
- 調査対象: 全国の小学校教員500名
- 主要な発見:
  - 約70%の学校で「完成見本を模倣する」形式の教材を使用
  - 教員の約60%が「児童がプログラミングを理解しているか不安」と回答
  - 「作品が完成すれば良い」という意識が強い

**模倣学習の構造的問題**:

1. **試行錯誤の欠如**
   - 手順書通りに進めれば必ず完成する設計
   - 「なぜその処理が必要か」を考える機会がない
   - エラーに遭遇する経験が少ない

2. **プログラミング的思考の未発達**[4]
   - 静岡大学の研究(2021年)による実証的知見:
     - 模倣型学習群と問題解決型学習群を比較
     - 論理的思考力テストの結果、問題解決型が有意に高得点
     - 模倣型では知識の転移(応用)が起こりにくい

3. **「できたつもり」問題**
   - 作品が完成するため達成感はある
   - しかし、仕組みを理解していないことが多い
   - 応用問題になると対応できない

**実際の教育現場での声**:
- 「子どもたちは完成させることに夢中で、なぜそうなるのか考えていない」
- 「見本と違う課題を出すと、何をしたらいいかわからなくなる」
- 「プログラミングができるようになったというより、手順を覚えただけ」

#### 3.1.4 発達段階と学習設計

**ピアジェの発達段階論との関連**[5]:
- **具体的操作期(7〜11歳)**:
  - 論理的思考が発達し始める時期
  - 具体的な事物に対しては論理的に考えられる
  - 抽象的な概念はまだ難しい

**小学3〜6年生が対象として適切な理由**:
- 3年生以上であれば基本的な論理演算(AND, OR等)が理解可能
- 段階的な学習設計が重要(いきなり複雑な課題は不適切)
- 学年に応じた支援(ヒント、説明)が必要

### 3.2 既存サービス・研究

#### 3.2.1 既存のプログラミング学習プラットフォーム

**1. Scratch公式サイト(scratch.mit.edu)**
- **概要**: MIT開発のビジュアルプログラミング環境
- **機能**:
  - ブラウザ上でプログラムを作成・実行
  - 作品の共有コミュニティ
  - チュートリアル動画
- **長所**:
  - 無料で誰でも利用可能
  - 世界中で広く使われている
  - 豊富な作品例
- **短所**:
  - 自動採点機能がない
  - 段階的な学習カリキュラムがない
  - 完全に自由制作なので、何を作るか迷う
  - 学習進捗の管理機能がない

**2. Code.org**
- **概要**: 非営利団体が提供する無料プログラミング教育プラットフォーム
- **機能**:
  - 年齢別のコース設計
  - ゲーム感覚で学べる課題
  - 自動チェック機能
- **長所**:
  - 段階的なカリキュラム
  - 即座のフィードバック
  - ゲーミフィケーション要素
- **短所**:
  - 主に英語コンテンツ(日本語は限定的)
  - Scratchではなく独自のブロック言語を使用
  - 既定のカリキュラムに沿うため柔軟性が低い
  - Scratch本体(.sb3)の提出・評価には対応していない

**3. Progate / その他オンライン学習サービス**
- **概要**: 主に中学生〜大人向けのプログラミング学習サイト
- **短所**(小学生にとって):
  - テキストベースのコーディングが中心
  - 小学校のScratch学習には対応していない
  - 難易度が高い

#### 3.2.2 自動採点に関する先行研究

**プログラミング自動採点の既存手法**:

1. **構文比較方式**
   - プログラムコードを文字列として比較
   - 正解コードと完全一致を求める
   - **問題点**: 変数名の違いや順序の違いで不正解になる

2. **実行結果比較方式**
   - プログラムを実行し、出力結果のみを比較
   - **問題点**: 正しい処理でなくても偶然正解になる可能性

3. **テストケース方式**
   - 複数の入力に対する出力をチェック
   - **問題点**: Scratchのようなビジュアル作品には適用困難

**本研究との違い**:
- **セマンティック(意味論的)比較**: ブロックの構造と意味を理解して比較
- **変数マッピング機能(v2.1)**: 変数名の違いを自動的に吸収
- **柔軟な評価**: ±10%の許容範囲を設けて小学生の入力誤差を考慮
- **詳細なフィードバック**: どこが間違っているか具体的に指摘

#### 3.2.3 問題ベース学習(PBL)との関連

**問題ベース学習(Problem-Based Learning)とは**:
- 学習者が問題を解決する過程で知識を習得する教育手法
- 「教えてから練習」ではなく「問題から学ぶ」
- 本研究の4種類の問題形式はPBLの考え方に基づく

**本システムとPBLの対応**:
- fill_blank: 構造化された問題解決
- predict: 予測と検証のサイクル
- find_error: デバッグという問題解決
- mission: 創造的問題解決

### 3.3 本研究の位置づけ

既存の研究・サービスと本研究の違いを整理すると:

| 観点 | Scratch公式 | Code.org | 本研究 |
|------|-----------|----------|--------|
| 自動採点 | ✗ | ○(独自言語) | ○(Scratch .sb3) |
| 変数名柔軟性 | - | - | ○(マッピング機能) |
| 段階的学習 | ✗ | ○ | ○(4タイプ問題) |
| 学年別対応 | ✗ | △ | ○(3〜6年生) |
| 模倣からの脱却 | △ | ○ | ○(設計思想) |
| 日本語完全対応 | ○ | △ | ○ |

**本研究の新規性**:
1. Scratch本体(.sb3ファイル)の自動採点システム
2. 変数名を無視する変数マッピング機能
3. 模倣学習から試行錯誤型学習への転換を支援する4タイプ問題設計
4. 学年別適応型ヒント機能

### 4.5 自動採点エンジン(v2.1)の詳細

#### 4.5.1 採点エンジンの設計方針
- セマンティック比較の重要性
- 変数名の違いを吸収する必要性

#### 4.5.2 変数マッピング機能
**概要**:
- 変数名を無視し、使用パターンで変数を自動マッピング
- 例: 正解「counter」↔生徒「りんご」→使用方法が同じなら正解

**アルゴリズム**:
```
1. 正解SB3と提出SB3から変数リストを抽出
2. 各変数の使用箇所・使用方法を解析
3. 使用パターンが一致する変数同士をマッピング
4. マッピング後のブロック構造を比較
```

#### 4.5.3 ブロック比較ロジック
- ブロックタイプの一致確認
- パラメータの柔軟な比較(±10%許容)
- 順序制約のチェック

#### 4.5.4 フィードバック生成
- 成功/警告/エラーの3段階
- 具体的なヒントの提示
- 詳細なフィードバック構造(summary, details[], hints[])

**フィードバックの例**:
```json
{
  "summary": "80点: 惜しい!あと少しです",
  "details": [
    {
      "type": "success",
      "message": "変数「カウンター」は正しく使われています"
    },
    {
      "type": "warning",
      "message": "繰り返しの回数が少し違います(期待値: 10, あなた: 9)"
    }
  ],
  "hints": [
    "繰り返しブロックの回数を確認してみましょう"
  ]
}
```

#### 4.6.2 ログイン画面

## 13. 想定される質問と回答の準備

### 13.1 技術面
**Q: なぜ変数名を無視する必要があるのか?**
A: 小学生は自由な発想で変数名を付けるため、名前の違いで不正解にするのは教育的に不適切。本質的な理解を評価するため。

**Q: ±10%の許容範囲の根拠は?**
A: 予備実験の結果、小学生の入力誤差を考慮した値。厳密すぎると学習意欲を削ぐ。

**Q: なぜScratchを選んだのか?**
A: 小学校で最も普及しているビジュアルプログラミング言語であり、.sb3形式が解析しやすいため。

### 13.2 教育面
**Q: 4種類の問題タイプの教育的根拠は?**
A: 計算論的思考の要素(分解・パターン認識・抽象化・アルゴリズム化)を段階的に学習するため。

**Q: 従来の学習方法と比較した利点は?**
A: 即座のフィードバック、個別最適化、教員負担軽減。
